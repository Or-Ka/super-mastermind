# יומן החלטות ארכיטקטוניות (ADR)

## ADR-001 — כל הנתונים והחישובים מקומיים בתהליך ה־Renderer

### סטטוס
התקבל

### הקשר
נדרשה שמירת הגדרות, סטטיסטיקות והיסטוריה מקומית, ועבודה מלאה ללא אינטרנט.
ב־Electron אפשר לשמור נתונים בתהליך הראשי (קבצים ב־userData דרך IPC) או
בתהליך ה־Renderer (localStorage/IndexedDB).

### חלופות שנשקלו
1. קבצי JSON בתהליך הראשי + שכבת IPC מלאה.
2. localStorage בתהליך ה־Renderer (ללא IPC).
3. SQLite / IndexedDB.

### החלטה
localStorage בתהליך ה־Renderer, מאחורי שכבת עטיפה (`src/storage/local.ts`).

### נימוקים
- היקף הנתונים קטן (הגדרות, עד 200 משחקים, עד 100 חידות) — הרחק מתחת למגבלות localStorage.
- מייתר פרוטוקול IPC שלם, preload מורחב וטיפול בשגיאות קבצים.
- ב־Electron הנתונים נשמרים בפרופיל האפליקציה ושורדים סגירה — עונה על הדרישה.
- שכבת העטיפה מרכזת את הגישה, כך שמעבר עתידי לקבצים/SQLite נקודתי.

### חסרונות
- הנתונים קשורים לפרופיל ה־Renderer (ניקוי "נתוני אתר" ימחק אותם).
- אין גיבוי קבצים ידידותי — ממותן ע"י ייצוא/ייבוא JSON ידני.

### השלכות
ייצוא/ייבוא JSON הוא מנגנון הגיבוי הרשמי. אם בעתיד יידרש סנכרון קבצים,
יוחלף המימוש בתוך `storage/local.ts` בלבד.

## ADR-002 — CSS מסודר עם משתני עיצוב במקום Tailwind

### סטטוס
התקבל

### הקשר
האפיון מתיר "CSS מסודר או Tailwind". נדרשות ערכות נושא (בהיר/כהה), RTL מלא,
שינוי גודל ממשק ומצב נגישות.

### חלופות שנשקלו
1. Tailwind CSS v4.
2. CSS Modules לכל רכיב.
3. קובץ CSS גלובלי מאורגן עם CSS Custom Properties.

### החלטה
קובץ CSS גלובלי מאורגן (`src/styles.css`) עם משתני `--*` לערכות נושא וגודל.

### נימוקים
- ערכות נושא וסקייל ממשק ממומשים טבעית עם משתנים על `:root`/`data-theme`.
- RTL מטופל גלובלית עם מאפיינים לוגיים (margin-inline וכו') — אין צורך בכלי נוסף.
- פחות תלויות ופחות צינור בנייה; ה־bundle קטן יותר.

### חסרונות
- אין utility classes — נדרשת משמעת בשמות מחלקות (נבחר BEM-רך).

### השלכות
כל צבע/מרווח/גופן עובר דרך משתנים. הוספת ערכת נושא = בלוק משתנים חדש.

## ADR-003 — הרצף הסודי שמור ב־closure של מנוע המשחק

### סטטוס
התקבל

### הקשר
האפיון דורש שהרצף הסודי לא יופיע ב־DOM ולא יהיה קל לחשוף אותו דרך הממשק.

### חלופות שנשקלו
1. שמירת הסוד ב־state של React (נחשף ב־React DevTools ובסריאליזציה).
2. שמירת הסוד בתהליך הראשי של Electron מאחורי IPC.
3. שמירת הסוד ב־closure בתוך `createGameEngine`, מוחזק ב־ref מחוץ ל־state.

### החלטה
חלופה 3.

### נימוקים
- הסוד אינו חלק מ־props/state ולכן לא מגיע ל־DOM ולא נצפה בכלי פיתוח רגילים.
- אין צורך בקריפטוגרפיה (לא נדרש באפיון) ולא בסיבוכיות IPC.
- `reveal()` נקרא רק בסיום משחק או במצב פיתוח (שכבוי ב־Production).

### חסרונות
- משתמש מתוחכם עם debugger יוכל לחלץ את הסוד — מקובל לפי האפיון.

### השלכות
שמירת משחק פעיל בין הפעלות אינה נתמכת (הסוד חי בזיכרון בלבד) — תועד במגבלות.

## ADR-004 — הרצת ה־Solver בתוך Web Worker

### סטטוס
התקבל

### הקשר
מנייה וסינון של עד מיליוני רצפים עלולות לחסום את ה־UI thread.

### חלופות שנשקלו
1. הרצה ישירה בתוך React (חוסם ממשק).
2. Web Worker.
3. הרצה בתהליך הראשי של Electron (IPC + חוסם את התהליך הראשי).

### החלטה
Web Worker יחיד (`src/workers/solver.worker.ts`) עם פרוטוקול הודעות מבוסס id.

### נימוקים
- הממשק נשאר מגיב; מוצג חיווי "מחשב..." בזמן עבודה.
- Vite תומך ב־Workers כ־module באופן מובנה (`new URL(..., import.meta.url)`).
- עובד זהה בדפדפן פיתוח וב־Electron, ללא רשת.

### חסרונות
- שכבת תקשורת אסינכרונית (ממותנת ב־hook `useSolverWorker` עם Promises).
- איתור שגיאות ב־Worker מעט מסורבל יותר.

### השלכות
כל קריאה נושאת id; תשובות מזדהות מול מפת Promises. ביטול = התעלמות מתשובה ישנה.

## ADR-005 — מנייה מדויקת עד תקרה, ומעבר לדגימה סטטיסטית

### סטטוס
התקבל

### הקשר
מרחב האפשרויות יכול להגיע ל־16^8 ≈ 4.3 מיליארד (16 צבעים, אורך 8, עם כפילויות).
מנייה מלאה שם אינה מעשית גם ב־Worker.

### חלופות שנשקלו
1. מנייה מלאה תמיד (נתקע במרחבים גדולים).
2. הגבלת ההגדרות כך שמרחבים גדולים לא יתאפשרו (פוגע בגמישות).
3. מנייה מלאה עד תקרה (1,000,000), ומעבר לה — דגימה אקראית אחידה (60,000) והערכה.

### החלטה
חלופה 3, עם דגל `estimated` שמוצג למשתמש ("בהערכה ~N").

### נימוקים
- בכל ההגדרות המעשיות (עד ~9 צבעים, אורך 4–5) המנייה מדויקת ומהירה.
- במרחבים קיצוניים המשתמש עדיין מקבל מידע שימושי בזמן סביר.
- עובדות לוגיות (רמזים) מדווחות **רק** במצב מדויק — הערכה לעולם לא תייצר רמז שגוי.

### חסרונות
- במצב מוערך אי אפשר להוכיח סתירה או פתרון יחיד — מוצג ניסוח זהיר בהתאם.

### השלכות
`AnalysisResult.estimated` מחלחל לכל הטקסטים בממשק ולרמת "ודאות הניתוח".

## ADR-006 — מחולל חידות מבוסס בחירת מועמדים לפי יעד צמצום

### סטטוס
התקבל

### הקשר
חידות "ניחוש מכריע" צריכות להיות עקביות, לא טריוויאליות, ובמצב "פתרון יחיד" —
להשאיר בדיוק פתרון אחד.

### חלופות שנשקלו
1. היסטוריה אקראית לגמרי (עקבית אך לרוב משעממת או בלתי פתירה בניסיון אחד).
2. סימולציה של פותר אופטימלי (Knuth) — החידות יוצאות דומות זו לזו.
3. בכל שלב: דגימת מועמדים (חציים מתוך הפתרונות האפשריים), ובחירת המועמד
   שמקרב את מספר הפתרונות ליעד מדורג (צמצום ~35% לשלב, יעד סופי לפי קושי).

### החלטה
חלופה 3, עם עד 8 נסיונות בנייה מחדש ונפילה בטוחה לחידה אקראית עקבית.

### נימוקים
- כל הציונים מחושבים מול סוד אמיתי ⇒ עקביות מובטחת מבנית (לפחות פתרון אחד).
- היעד המדורג מייצר עקומת צמצום "אנושית" ומעניינת.
- מצב פתרון־יחיד ממומש בהוספת ניחושים עד שנותר בדיוק אחד, עם בדיקת אימות סופית.

### חסרונות
- יצירת חידה קשה אורכת עד ~3 שניות — רצה ב־Worker עם חיווי טעינה.

### השלכות
בדיקות אוטומטיות מאמתות: עקביות, ≥1 פתרון, ובדיוק 1 במצב פתרון־יחיד.
